## 1. テーマ別ハイライト
### T1 MoE / 効率的アーキテクチャ
今週のT1は、モデル本体の大規模刷新よりも「既存モデルをPCIe環境でどう速く・長く・安く回すか」に重心があった。`TAPS` は workload-aware な speculative decoding を示し、金融QAと汎用対話で drafter を分ける実装方針に直結する。`Universal YOCO` や `PackForcing` は KV cache と長文脈コストの抑制に焦点があり、長い社内文書や多段エージェント履歴を抱える運用条件と噛み合う。`Brainstacks` は MoE-LoRA の継続学習を量子化前提で扱っており、複数の金融サブドメインLoRAを後付け合成する現実路線として面白い。総じて、今週のT1は「推論速度・履歴圧縮・継続拡張」を一体で考える週だった。

### T2 金融AI / ドメイン特化LLM
T2は、金融そのものの専用論文が少ない一方で、金融RAGや金融文書処理にそのまま刺さる評価・検索・文書理解の論文が厚かった。`FinMCP-Bench` が金融MCPを前提にした実運用ベンチを提示し、金融エージェント評価の基準線がかなり具体化した。`When Documents Disagree`、`Working Notes on Late Interaction Dynamics`、`MDPBench` は、文書間不整合、長文バイアス、多言語文書パースというRAGの実務ボトルネックを直接突いている。さらに `SEAR` は構造化スキーマで品質・遅延・コストを同時監視する gateway 設計を示し、構造化出力品質が最重要課題である現状と一致する。T2の流れは、「より強いモデル」より「文書基盤・評価基盤・監査基盤を作る」方向へ明確に寄っている。

### T3 マルチエージェント / ツール利用
今週もっとも厚かったのはT3で、研究の主戦場が agent loop そのものから運用基盤へ移っていることがはっきり見えた。`Trace2Skill`、`SKILL0`、`Natural-Language Agent Harnesses` は、スキルやハーネスを外部アーティファクトとして整理し、必要なら最終的に内在化する流れを示した。`Marco DeepResearch`、`MiroEval`、`Signals`、`Meta-Harness` は、探索・検証・ログ選別・harness最適化まで含めて agent を評価し直しており、最終回答だけでは不十分という認識が強い。`Terminal Agents Suffice for Enterprise Automation` や `ToolRosetta` は、複雑なGUIや抽象化より API/terminal-first と MCP 標準化が実務的だと示した。T3は今週、「エージェントを賢くする」から「エージェント運用を設計する」へ進んだ。

### T4 ファインチューニング / アライメント
T4は、重いRLを回すかどうか以前に、何を学習信号にし、どう監査し、どうデータを制御するかに焦点が集まった。`DataFlex` が sample selection・mixture adjustment・reweighting を統合し、データ品質の改善がまず効くことを強く示した。`FIPO`、`Apriel-Reasoner`、`SKILL0` は、出力全体ではなく重要トークンやスキル内在化に寄せた学習設計で、長い思考をただ増やすより効率的な後学習を志向している。`MonitorBench` と `Reasoning Shift` は、説明やCoTがそのまま監査可能性を保証しないことを示し、金融用途では「説明したか」ではなく「意思決定因子を追えているか」を見る必要がある。T4全体としては、重み更新そのものよりもデータ制御・監査可能性・低遅延化が前面に出た週だった。

## 2. テーマ横断マップ
今週の横断トレンドは、「制御層の強化」でほぼ一貫している。T1の speculative decoding・KV制御・文脈圧縮は、T3の harness 最適化や skill orchestration と組み合わさることで、限られたPCIe環境でも実用レイテンシを出す方向に接続する。T2の文書不整合評価、late interaction解析、多言語document parsingは、T4の MonitorBench や Reasoning Shift と繋がり、RAGで根拠を拾うだけでなく、その根拠が最終判断にどう効いたかまで監査する必要性を押し上げている。さらに T3 の `Trace2Skill`・`SKILL0` と T4 の `DataFlex` は、運用ログや手順書をそのまま資産化し、外部スキル注入から内在化へ移す流れを共有している。要するに今週は、「強い基盤モデルを探す週」ではなく、「既存モデルを安全に・速く・監査可能に・継続改善可能に運用する方法」が全テーマで収束した。

## 3. 金融AIエージェントへの示唆
### 今週のアクションアイテム（最大3つ）
1. `FinMCP-Bench` を雛形に、社内MCP・単一/複数ツール・マルチターンを含む金融エージェント評価セットを作る。
2. `DataFlex` と `Signals` を参考に、SFT/DPO用データを全量投入せず、運用ログから高情報価値サンプルを選別するデータ制御パイプラインを作る。
3. `SEAR`、`When Documents Disagree`、`MonitorBench` を組み合わせ、構造化出力品質・根拠欠落・説明の監査可能性を同時に測る評価指標を整備する。

### 中期的に注視すべき方向
agent の性能改善は、今後はモデル本体よりも harness・memory・verification・skill internalization の設計差で決まりやすい。金融領域では特に、terminal/API-first な実行系、長文書RAGの監査、運用ログからの継続改善が競争力の中心になる。MoE/量子化基盤については、新規アーキテクチャ探索より speculative decoding、KV管理、LoRA/MoEスタックの運用最適化を先に詰める方が費用対効果が高い。

### 今週スキップしてよい理由
純粋な汎用 reasoning 改善だけを狙う重いRL路線は、今週の流れを見る限り優先度を上げすぎなくてよい。理由は、現状のボトルネックがモデル知能の絶対値より、評価基盤不足、データ品質不足、文書監査不足、運用制御不足にあるからだ。まず制御層を整えた方が、同じ基盤モデルでも成果が出やすい。
