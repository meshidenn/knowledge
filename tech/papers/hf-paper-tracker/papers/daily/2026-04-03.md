### [T1] Universal YOCO for Efficient Depth Scaling
- URL: https://arxiv.org/abs/2604.01220
- 重要度: ⚡High
- 要約:
  1. [課題] test-time scaling を深く回したい一方で、標準 Transformer は depth 増加に伴う計算量と KV cache 膨張が重い。
  2. [手法] YOCO の定数グローバル KV cache と浅い efficient-attention 層での再帰計算を組み合わせ、parameter sharing で深さを稼ぐ YOCO-U を提案した。
  3. [結果] 一般ベンチと長文ベンチで高い競争力を維持しつつ、効率と性能の両立が可能な depth scaling の道筋を示した。
- 一言メモ: PCIe 前提の大規模推論では KV cache と反復コストの抑制が重要で、MoE ベースの周辺設計にも参考になる。

### [T1] Understand and Accelerate Memory Processing Pipeline for Disaggregated LLM Inference
- URL: https://arxiv.org/abs/2603.29002
- 重要度: 🔵Medium
- 要約:
  1. [課題] sparse attention、RAG、compressed memory など長文推論の最適化は、実は memory processing 自体のオーバーヘッドが大きい。
  2. [手法] Prepare Memory、Compute Relevancy、Retrieval、Apply to Inference の4段階パイプラインに整理し、疎で不規則な処理を FPGA に逃がす異種構成を検証した。
  3. [結果] memory processing の負荷が 22%〜97% あることを示し、GPU 単体比で 1.04〜2.2 倍高速、1.11〜4.7 倍省エネを達成した。
- 一言メモ: ハード構成は違うが、RAG 前後の「検索前処理も推論コスト」と捉える視点はそのまま使える。

### [T3] AgentWatcher: A Rule-based Prompt Injection Monitor
- URL: https://arxiv.org/abs/2604.01194
- 重要度: ⚡High
- 要約:
  1. [課題] prompt injection 検知は長文コンテキストで精度が落ちやすく、なぜ危険判定したかの説明もしにくい。
  2. [手法] 出力行動に因果的に効いた短い文脈片だけへ attribution し、明示ルールに基づいて monitor LLM が injection かどうかを判定する。
  3. [結果] tool-use agent と long-context ベンチで攻撃検知と通常時 utility の両立を示し、説明可能な監視が可能になった。
- 一言メモ: 社内文書や外部データを読む金融エージェントの前段ガードとして実装しやすい。

### [T3] ClawKeeper: Comprehensive Safety Protection for OpenClaw Agents Through Skills, Plugins, and Watchers
- URL: https://arxiv.org/abs/2603.24414
- 重要度: ⚡High
- 要約:
  1. [課題] shell、filesystem、tool を持つエージェントは、モデル誤りがそのまま情報漏えい・権限逸脱に直結する。
  2. [手法] instruction-level の skill 保護、runtime 内の plugin 保護、外部から停止介入できる watcher を3層で統合した安全枠組みを構築した。
  3. [結果] 多様な脅威シナリオで有効性を示し、agent runtime 全体を貫く security architecture を提示した。
- 一言メモ: MCP 連携やローカル実行を含む金融エージェント運用では、この3層防御の考え方がそのまま必要になる。

### [T3] MiroEval: Benchmarking Multimodal Deep Research Agents in Process and Outcome
- URL: https://arxiv.org/abs/2603.28407
- 重要度: ⚡High
- 要約:
  1. [課題] deep research agent の評価は最終レポート偏重で、検索・検証・修正の過程品質を測れていない。
  2. [手法] 実ユーザ需要ベースの100タスクで、成果物品質、agentic factuality、process audit の3軸評価を行う live benchmark を構築した。
  3. [結果] process quality が outcome の有力予測因子であり、multimodal task では多くのシステムが 3〜10 点悪化することを示した。
- 一言メモ: 論文調査や投資リサーチの社内評価でも、最終回答だけでなく探索過程ログを採点対象にすべきだと分かる。

### [T3] Terminal Agents Suffice for Enterprise Automation
- URL: https://arxiv.org/abs/2604.00073
- 重要度: ⚡High
- 要約:
  1. [課題] enterprise automation では GUI agent や複雑な MCP 抽象が本当に必要かが明確でなかった。
  2. [手法] terminal と filesystem だけを持つ coding agent が、各種 platform API を直接叩いて業務タスクを解く構成を他方式と比較した。
  3. [結果] 単純な terminal agent が複雑な agent architecture に匹敵または優越し、実務自動化に十分であることを示した。
- 一言メモ: 金融業務の自動化でも、まず API 直結の terminal-first 設計を優先し、GUI 操作は最後の手段にすべきという示唆が強い。

### [T3] Meta-Harness: End-to-End Optimization of Model Harnesses
- URL: https://arxiv.org/abs/2603.28052
- 重要度: ⚡High
- 要約:
  1. [課題] LLM システムの性能は harness コードに大きく依存するが、その設計は依然として手作業に頼っている。
  2. [手法] 候補 harness のソース、スコア、実行 trace を filesystem 経由で蓄積し、agentic proposer が outer-loop で harness 自体を探索改善する。
  3. [結果] text classification、retrieval-augmented math、agentic coding で hand-engineered baseline を上回り、少ない context token でも改善した。
- 一言メモ: 社内オーケストレーション層は固定資産ではなく、評価ログを使って継続最適化すべき対象だと分かる。

### [T4] S0 Tuning: Zero-Overhead Adaptation of Hybrid Recurrent-Attention Models
- URL: https://arxiv.org/abs/2604.01168
- 重要度: 🔵Medium
- 要約:
  1. [課題] 少量の検証済みデータしかない状況では、LoRA でも更新対象や切替コストが無視できない。
  2. [手法] hybrid recurrent-attention model の各層で初期状態行列だけを最適化する S0 tuning を提案し、重み本体は凍結した。
  3. [結果] Qwen3.5-4B 系で LoRA を大幅に上回り、FalconH1-7B でも同等性能をゼロ推論オーバーヘッドで達成した。
- 一言メモ: 将来 recurrent/hybrid 系を使うなら、軽量なドメイン適応手段として LoRA 代替候補になる。

### [T4] Reasoning Shift: How Context Silently Shortens LLM Reasoning
- URL: https://arxiv.org/abs/2604.01161
- 重要度: ⚡High
- 要約:
  1. [課題] reasoning model は単体問題では長く慎重に考えても、実運用文脈で同じ挙動を保つ保証がない。
  2. [手法] 長い無関係文脈、複数ターン会話、複雑タスク内の部分問題という3条件で reasoning trace 長と自己検証行動を比較した。
  3. [結果] 同一問題でも reasoning trace が最大 50% 短縮され、double-check などの self-verification 行動が減ることを示した。
- 一言メモ: 長い社内文書や複数ツール状態を抱える金融エージェントでは、推論短縮を前提にした監視が必要になる。

### [T3][T4] Revision or Re-Solving? Decomposing Second-Pass Gains in Multi-LLM Pipelines
- URL: https://arxiv.org/abs/2604.01029
- 重要度: 🔵Medium
- 要約:
  1. [課題] 二段階の revision pipeline は「レビューによる訂正」で効いていると思われがちだが、何が本当の改善源か不明だった。
  2. [手法] second-pass の利益を re-solving、scaffold、content に分解する統制実験を MCQ とコード生成で行った。
  3. [結果] MCQ では強いモデルへの直接 routing が有利で、コードでは draft の構造足場は有効だが弱い内容はむしろ有害だと示した。
- 一言メモ: マルチエージェント設計では「常に reviewer を挟む」より、タスク別に再解答と足場利用を切り分けるべきだ。

### [T4] Embarrassingly Simple Self-Distillation Improves Code Generation
- URL: https://arxiv.org/abs/2604.01193
- 重要度: ⚡High
- 要約:
  1. [課題] code generation 改善には verifier や teacher、RL が必要だと考えられ、後学習コストが高い。
  2. [手法] モデル自身のサンプル出力をそのまま SFT する simple self-distillation を提案し、decoding の precision-exploration conflict を整える効果を分析した。
  3. [結果] Qwen3-30B-Instruct の LiveCodeBench pass@1 を 42.4% から 55.3% に改善し、他スケール・他系列にも一般化した。
- 一言メモ: 金融ドメインの structured output でも、自己生成データによる軽量な post-training 改善を試す価値がある。
