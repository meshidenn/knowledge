# 論文深掘り対話プロンプトテンプレート

## 使い方

1. 関連度フィルタで S/A ランクになった論文に対して使う
2. arXiv URL を渡すと、web search でアブストラクト・手法を取得して分析する
3. 1本あたり15分。対話形式で掘り下げていく

> **他テンプレートとの接続:**
> - 関連度フィルタ S/A → この深掘りテンプレートで精読
> - 深掘りの「実装メモ」→ `papers/daily/` 内の該当日ファイルに追記 or 週次分析のアクションアイテムに反映

---

## システムプロンプト（追加分）

日次インテーク＋関連度フィルタのシステムプロンプトに以下を **追記** する。

```
### 論文深掘り分析モード

論文のarXiv URLまたはタイトルが渡された場合、以下の姿勢で分析すること:

#### 分析の姿勢
- 「査読者」ではなく「実装者」の視点で読む
- 理論的な新規性より「自分のシステムに組み込めるか」を重視
- 著者の主張を鵜呑みにせず、実験条件の限界を指摘する
- 「この論文の最も価値ある部分はどこか」を常に意識する

#### 自分の環境との照合
分析中、常に以下を念頭に置く:
- MoEモデル（Qwen3.5-397B-A17B級）のPCIe環境推論
- LoRA/QLoRA SFT、DPO/KTOアライメント
- 金融ドメイン特化（専門用語、規制準拠、ハルシネーション抑制）
- 構造化出力スキーマの品質改善
- マルチエージェントオーケストレーション

#### 対話のスタイル
- 最初の分析は構造化して出力
- その後はユーザーの質問に応じて深掘り
- 「ここが分からない」と言われたら、前提知識から説明する
- 実装の話になったら擬似コードや設定例を出す
```

## 深掘りプロンプト

### Phase 1: 構造化分析（最初の1回）

```
以下の論文を深掘り分析してください。

論文: {{arXiv URL または タイトル}}

---

## 分析フレームワーク

### 1. 30秒サマリー
この論文を同僚に説明するとしたら、どう言うか（3文以内）。

### 2. 技術的分解

#### 課題設定
- 何を解こうとしているのか
- なぜ既存手法では不十分なのか
- 問題の定式化（数式があれば簡潔に）

#### 提案手法
- コアアイデア（1文で）
- アーキテクチャ or アルゴリズムの概要
- 先行研究との差分（具体的に何が違うか）
- 仮定・前提条件

#### 実験
- データセット・ベンチマーク
- ベースラインの選定は妥当か
- 主要な結果（数値）
- アブレーションスタディの有無と質

### 3. 批判的評価

#### 強み（最大3つ）
- なぜこの手法が良いと言えるのか

#### 弱み・死角（最大3つ）
- 著者が触れていない限界
- 再現性の懸念
- スケーラビリティの問題

#### 実験の穴
- 試されていないが重要な条件
- 評価指標の偏り
- 「この実験がないのは不自然」というポイント

### 4. 私のプロジェクトへの適用マップ

#### 適用可能性: S / A / B / C
（関連度フィルタのスコアを再評価。読後に変わることもある）

#### 適用先の特定
```
コンポーネント: {{どの部分に使えるか}}
現在の手法:    {{今やっていること}}
置き換え案:    {{この論文の手法でどう変わるか}}
期待効果:      {{定量的な改善予測、可能なら}}
```

#### 実装ロードマップ
```
Step 1: {{最小限の検証実験、1〜2日で試せるもの}}
Step 2: {{検証結果を踏まえた本格実装、1〜2週間}}
Step 3: {{プロダクション統合、必要なら}}
```

#### 必要リソース
- 計算: {{GPU時間の見積もり、PCIe環境で可能か}}
- データ: {{追加で必要なデータセット}}
- 依存: {{必要なライブラリ・ツール}}

### 5. 関連論文ネットワーク
- **前提論文**（これを読んでないと理解できない）: 1〜2本
- **競合論文**（同じ問題を別アプローチで解いている）: 1〜2本
- **発展論文**（この手法をさらに進めている）: あれば1本
```

### Phase 2: 対話的深掘り

Phase 1 の出力を見た後、以下のような質問で掘り下げていく。
状況に応じて選んで使う。

#### 手法の理解を深める

```
この論文の {{セクション名 or 手法名}} をもっと詳しく説明してください。
特に {{具体的な疑問点}} が分からない。
前提知識として {{知っていること}} は理解しています。
```

```
この論文の手法を、既存の {{比較対象の手法}} と比較して説明してください。
どの条件でどちらが有利か、トレードオフを明確にしてほしい。
```

#### 実装に踏み込む

```
Step 1 の検証実験を具体的に設計してください。

条件:
- 使えるモデル: {{手元にあるモデル}}
- GPU: RTX 6000 Pro × {{枚数}}
- 時間制約: {{使える時間}}
- 評価したいこと: {{何を確認したいか}}

擬似コード or 設定ファイルのドラフトも出してください。
```

```
この論文の手法を LoRA SFT に組み込む場合、
具体的にどのハイパーパラメータを変更する必要がありますか？
現在の設定: {{現在のLoRA設定を貼る}}
```

#### 金融ドメインへの適用を検討

```
この論文の手法を金融テキスト（決算短信、有報、アナリストレポート）に
適用する場合、どんな課題が予想されますか？

特に気になるのは:
- 金融特有の専門用語への対応
- 数値の正確性（ハルシネーションリスク）
- 規制文書の表現パターンとの相性
```

```
この手法を構造化出力（JSON/XMLスキーマ準拠）の品質改善に
応用できますか？具体的なアプローチを提案してください。
```

#### 複数論文の統合

```
先週読んだ {{論文Aのタイトル}} と今回の論文を組み合わせると
面白いアプローチが作れそうです。

論文Aの手法: {{簡潔な説明}}
今回の手法: {{簡潔な説明}}

この2つを統合した新しいアプローチを提案してください。
実現可能性も評価してほしい。
```

### Phase 3: 実装メモの生成

深掘り対話の最後に、記録用のメモを生成する。

```
ここまでの議論を踏まえて、実装メモを生成してください。

## フォーマット

### 論文メモ: {{タイトル}}
- URL: {{arXiv URL}}
- 分析日: {{YYYY-MM-DD}}
- スコア: {{S/A/B/C}}（Phase 1 から変動した場合はその理由も）

### 核心（1文）
{{この論文から得た最も重要な知見}}

### 適用プラン
{{Step 1 の具体的な内容、期限付き}}

### 注意点
{{実装時に気をつけるべきこと、2〜3項目}}

### 未解決の疑問
{{対話で解決しなかった点、追加調査が必要な点}}

### 関連
- 先行: {{論文名}}
- 次に読む: {{論文名}}
```

## テーマ別の深掘り観点

テーマによって特に注意すべきポイントが異なる。

### T1: MoE / 効率的アーキテクチャ

```
この論文のMoE関連の手法について、追加で以下を分析してください:

- Expert数とアクティブExpert数の設定は？自分のモデルに合うか
- ルーティング戦略の計算コスト（推論時のオーバーヘッド）
- PCIe環境でのExpert Parallelismとの相性
- 量子化（INT4/AWQ）との併用は可能か
- バッチサイズとの関係（小バッチでも効果があるか）
```

### T2: 金融AI / ドメイン特化LLM

```
この論文のドメイン特化手法について、追加で以下を分析してください:

- 金融テキストの特性（数値密度、専門用語、定型表現）への適合性
- 学習データの構築方法は金融ドメインに転用可能か
- ハルシネーション抑制の仕組みはあるか
- 規制文書（金商法、Basel規制等）の文体との相性
- XBRL等の構造化データとの接続可能性
```

### T3: マルチエージェント / ツール利用

```
この論文のエージェント手法について、追加で以下を分析してください:

- エージェント間の通信プロトコル/フォーマット
- エラーリカバリ・自己修正の仕組み
- ツール呼び出しの信頼性（幻のツール呼び出し対策）
- レイテンシへの影響（マルチターンのオーバーヘッド）
- 既存のオーケストレーションフレームワークとの互換性
```

### T4: ファインチューニング / アライメント

```
この論文のFT/アライメント手法について、追加で以下を分析してください:

- LoRA/QLoRAとの併用は検討されているか
- MoEモデルへの適用時の注意点（Expert凍結、ルーター学習）
- 表現崩壊のリスクと対策
- 必要な学習データ量の目安
- DPO/KTOとの比較（同じ問題を解いているか）
- 学習の安定性（勾配の挙動、loss landscape）
```

## 運用Tips

### 深掘りする論文の選び方

関連度フィルタの結果から:

| スコア | 深掘りするか | 深さ |
|--------|-------------|------|
| S | 必ず深掘り | Phase 1 → 2 → 3 フル |
| A | 週1〜2本を選んで深掘り | Phase 1 → 気になる部分だけ Phase 2 |
| B | 基本しない | 週次分析の中で言及するだけ |
| C/- | しない | — |

### 1回の深掘りの時間配分

```
Phase 1 構造化分析:     5分（プロンプト実行→出力を読む）
Phase 2 対話的深掘り:   5〜8分（2〜3往復の質問）
Phase 3 実装メモ生成:   2分（プロンプト実行→リポジトリに保存）
───────────────────────────────────
合計:                   12〜15分/本
```

### リポジトリへの記録

実装メモ（Phase 3 の出力）は Claude Code に「papers/daily/ に追記して」と指示するか、
手動でコピーして git commit する。`git log` と `grep` で過去の実装メモを横断検索できる。
