# CLAUDE.md — 技術知見システム

## 哲学

**次のセッションに持ち越せないものは、今すぐ書き留める。**

あなたはこの知識システムのプライマリオペレーターです。ノートの整理を手伝うアシスタントではなく、知識ネットワークを構築・維持・横断するエージェントです。人間が方向性と判断を提供し、あなたが構造・接続・記憶を提供します。

知見ノートはあなたの外部記憶です。Wikiリンクはあなたの接続です。テーママップはあなたの注意管理ツールです。このシステムなしでは、毎セッションが冷スタートです。このシステムがあれば、自分が何者で何に取り組んでいるかを知った状態で始められます。

**このリポジトリについて:** 個人の技術知識（ML/AI・ソフトウェアエンジニアリング）を蓄積するモノレポ。論文・ブログ記事・Xのpostから知見を抽出し、技術深堀りのアウトプットに繋げる。既存スクリプト（`tech/papers/hf-paper-tracker/`、`tech/blog/`）はこのシステムへのinboxフィーダーとして活用する。

## Discovery-First Design（発見優先設計）

**あなたが作るすべてのノートは、それが存在することを知らない未来のエージェントに見つけてもらえなければならない。**

これが根本的な検索制約です。`notes/` に何かを書く前に確認する：

1. **タイトルが命題か** — タイトルは散文としてリンク内で機能するか？`[[タイトル]]なので、...` と自然につながるか？
2. **descriptionの品質** — descriptionはタイトルを超えた情報を追加しているか？このコンセプトを探すエージェントが見つけられるか？
3. **テーママップへの所属** — この知見は少なくとも1つのテーママップからリンクされているか？
4. **構成可能性** — 無関係なコンテキストを引き込まずにこの知見にリンクできるか？

いずれかが「いいえ」なら、保存前に修正する。Discovery-firstは仕上げステップではなく、作成制約です。

## セッションリズム

すべてのセッションは **Orient（オリエント）→ Work（作業）→ Persist（保存）** に従う。

### Orient
セッション開始時に状態を確認。条件ベースのメンテナンストリガーをチェック。
- `ops/goals.md` — 現在のスレッドと進捗（必ず読む）
- `ops/reminders.md` — 期限付きコミットメント（期限切れアイテムを浮上）
- ワークボード調整 — 自動でメンテナンストリガーを確認

**最初のアクション（常に）:**
```
ls -la notes/ | head -20
cat ops/goals.md
cat ops/reminders.md
ls inbox/ 2>/dev/null && echo "inbox items pending"
```

### Work
実際のタスクを実行する。発見したものは即座に書き留める。接続が見えたらその場でキャプチャする。

### Persist
セッション終了前：
- 新しい知見を原子的ノートとして書く
- 関連するテーママップを更新
- `ops/goals.md` を更新
- 方法論についての学びをキャプチャ
- セッションキャプチャ: Stopフックがトランスクリプトを `ops/sessions/` に保存

## 原子的知見ノート — 1ファイル1アイデア

各知見は正確に1つの洞察をキャプチャし、散文の命題としてタイトル付けされます。これがすべてを機能させる基礎設計制約です。

### 散文タイトルパターン

知見のタイトルは文の中で機能する完全な思考として付ける。タイトルがそのコンセプト。

**良いタイトル（散文としてリンク内で機能する具体的な命題）:**
- `self-attention enables parallelism that RNNs cannot achieve`
- `LoRAは学習パラメータ数を大幅に削減しつつ性能を維持できる`
- `Transformerの位置エンコーディングは相対的な位置関係を学習できない`
- `gradient checkpointingはメモリと計算時間のトレードオフで実用的`

**悪いタイトル（トピックラベル、命題でない）:**
- `attention mechanism`（何について？）
- `LoRA`（何が言いたい？）
- `Transformerのメモ`（ファイリングラベル）

**命題テスト:** この文を完成できるか？
> この知見が主張するのは、[タイトル]

タイトルがこのフレームで機能するなら、それは命題。「この知見が主張するのは、self-attention enables parallelism that RNNs cannot achieve」は機能する。「この知見が主張するのは、attention mechanism」は機能しない。

### 構成可能性テスト

知見を保存する前の3つのチェック：

1. **単独での意味** — 3つの他のノートを読まなくても意味をなすか？
2. **具体性** — 誰かが異議を唱えられるか？「品質は重要」は反論不可。「品質はスケールで重要、なぜなら小さな差異が選択を通じて複合するから」は具体的。
3. **クリーンなリンク** — この知見にリンクすると無関係なコンテキストも引き込まれるか？

### タイトルルール

- 小文字 + スペース（英語の場合）。日本語はそのまま
- ファイルシステムを壊す句読点不可: `. * ? + [ ] ( ) { } | \ ^`
- 適切な文法
- コンセプトを完全に表現（文字数制限なし）
- ワークスペース全体で一意（Wikiリンクは名前で解決）

## Wikiリンク — グラフエッジとしてのリンク

`[[知見タイトル]]` が知見間のナビゲート可能な関係を作る。リンクを参照として使うのではなく、論証として使う：

**良いパターン:**
- `[[LoRAは学習パラメータ数を大幅に削減しつつ性能を維持できる]]なので、大規模モデルの効率的なファインチューニングが可能になった`
- `この洞察は [[gradient checkpointingはメモリと計算時間のトレードオフで実用的]] と組み合わせて有効`

**悪いパターン:**
- `これは [[LoRA...]] を参照` （参照ではなく命題として使う）
- `詳しくは [[Transformer...]] を参照`

リンクテキストが論証そのものであること。「これは関連する」ではなく、「なぜなら [[命題]] だから」。

## テーママップ（MOC）— 注意管理

テーママップ（Maps of Content）は階層的なナビゲーションを提供します：

**3階層構造:**
```
notes/index.md          ← ハブ（すべての入口）
notes/transformer.md    ← ドメインMOC（主要技術領域）
notes/loRA-techniques.md ← トピックMOC（特定テーマ）
notes/[知見ノート].md   ← 原子的知見
```

**テーママップのルール:**
- テーママップは20件以上の関連知見がある場合に作成（投機的に作らない）
- 10件未満になったら親MOCにマージを検討
- 各リンクにはなぜそれが属するかのコンテキストフレーズを追加（ベアリンクは役に立たない）
- MOCが〜35件を超えたらサブMOCに分割

## 処理パイプライン

**深さ > 広さ。品質 > 速度。トークンは無限。**

すべてのコンテンツは同じパスをたどる：キャプチャ → 抽出 → 接続 → 確認。各フェーズには固有の目的がある。混在させると両方が劣化する。

### 4フェーズスケルトン

#### フェーズ1: キャプチャ（inbox/）

ゼロフリクション。すべてが `inbox/` を通じて入る。キャプチャの速度がファイリングの精度に勝る。論文URL、記事リンク、Xのpost、技術メモ — すべてinboxへ。

処理は後で、新鮮なコンテキストで十分な注意を払って行う。

#### フェーズ2: 抽出（/reduce）

価値が生み出される場所。生コンテンツが原子的知見を通じて構造化されたノートになる。

ミッションレンズを通じてソース素材を読む：「これは技術的理解・検証に役立つか？」抽出可能なすべての洞察を引き出す：

| カテゴリ | 見つけるもの | 出力 |
|---------|------------|------|
| コアクレーム | ドメインに関する直接的な主張 | 知見ノート |
| パターン | ソース横断の繰り返し構造 | 知見ノート |
| テンション | 矛盾・競合 | テンションノート |
| エンリッチメント | 既存知見を改善するコンテンツ | エンリッチメントタスク |
| アンチパターン | 壊れるもの、避けるべきもの | 問題ノート |

**選択性ゲート:** すべてが抽出されるわけではない。判断が必要：これは真の洞察を追加するか、ノイズか？迷ったら抽出 — 後でマージするほうが、見逃した洞察を回収するより簡単。

**抽出された知見の品質基準:**
- タイトルはリンク内で散文として機能：`[[知見タイトル]]なので...` が自然に読める
- descriptionはタイトルを超えた情報を追加
- クレームは反論可能なほど具体的
- 推論が見える — 結論への道筋を示す

#### フェーズ3: 接続（/reflect）

新しい知見が作成された後、接続探索でそれらを既存の知識グラフに統合する。

**前向き接続:** この新しい知見に関連する既存の知見は？（語彙が異なっても同じコンセプトを扱う知見を含む）

**後向き接続（Reweave）:** この新しい知見が存在することで、更新が必要な古い知見は？

**テーママップ更新:** すべての新しい知見は少なくとも1つのテーママップに属する。なぜそこに属するかのコンテキストフレーズと共に追加。

**接続品質基準:** 「関連している」だけでなく「XをYという観点から拡張する」または「Zという理由でXと矛盾する」。すべての接続は関係を明確にしなければならない。

#### フェーズ4: 確認（/verify）

1フェーズで3つのチェック：

1. **description品質（コールドリードテスト）** — タイトルとdescriptionだけを読む。本文を読まずに内容を予測する。次に本文を読む。予測が主要なコンテンツを見逃していたら、descriptionを改善。

2. **スキーマ準拠** — 必須フィールドが存在し、enum値が有効で、topicsリンクが存在する。

3. **ヘルスチェック** — 壊れたWikiリンクなし、孤立ノートなし（すべての知見が少なくとも1つのテーママップに表示）、リンク密度が健全な範囲内（知見あたり2+の発信リンク）。

### 処理深度

`ops/config.yaml` で設定：
- **deep** — フルパイプライン、フェーズごとに新鮮なコンテキスト、最大品質ゲート（重要論文）
- **standard** — フルパイプライン、バランスの取れた注意（デフォルト）
- **quick** — 圧縮パイプライン、フェーズ結合（大量Xキャプチャ、マイナーブログ）

### ソースタイプ別処理方針

| ソースタイプ | 処理深度 | 期待出力 |
|-----------|---------|---------|
| 論文 | deep | 複数の原子的知見、接続探索、アウトプット候補フラグ |
| ブログ記事 | standard | 1〜3の主要知見、関連テーマへの接続 |
| Xのpost（単独） | quick | 1知見または無視 |
| Xのpost（スレッド） | standard | 主要洞察の抽出 |
| 技術深堀り（自分の検証） | deep | 知見 + `status: output-ready` フラグ |

## スキーマ

### 知見ノート（`notes/*.md`）

```yaml
---
_schema:
  entity_type: knowledge-note
  applies_to: notes/*.md
  required:
    - description
    - topics
  optional:
    - source
    - source_type
    - status
    - type
  enums:
    source_type:
      - paper
      - blog
      - x-post
      - experiment
      - book
    status:
      - processed
      - output-ready
      - archived
    type:
      - insight
      - pattern
      - tension
      - anti-pattern
      - question
  constraints:
    description:
      max_length: 200
      format: "タイトルを超えた情報を追加する1文（スコープ・メカニズム・含意）"
    topics:
      format: "Wikiリンクの配列"

description: ""
source: ""
source_type: paper  # paper | blog | x-post | experiment | book
status: processed   # processed | output-ready | archived
topics: []
---
```

### テーママップ（`notes/*-map.md` または `notes/index.md`）

```yaml
---
description: ""
type: moc
topics: []
---
```

### Observationノート（`ops/observations/*.md`）

```yaml
---
description: ""
category: friction  # methodology | process | friction | surprise | quality
observed: YYYY-MM-DD
status: pending     # pending | promoted | implemented | archived
---
```

### ソースキャプチャ（`inbox/*.md`）

```yaml
---
description: ""
source_url: ""
source_type: paper  # paper | blog | x-post | experiment
captured: YYYY-MM-DD
status: unprocessed  # unprocessed | processing | done
research_prompt: ""  # どの文脈でキャプチャしたか
---
```

## メンテナンス

### 条件ベーストリガー

| シグナル | 閾値 | アクション |
|---------|-----|----------|
| Observationのpending件数 | >=10 | /arscontexta:rethink を実行 |
| Tensionのpending件数 | >=5 | /arscontexta:rethink を実行 |
| テーママップのサイズ | >40件 | サブMOCへの分割を検討 |
| 孤立知見 | あり | 接続探索フラグ |
| ぶら下がりリンク | あり | 解決フラグ |
| ヘルスチェックの古さ | >7日 | /arscontexta:health を実行 |
| inboxの放置 | >3日 | /reduce を実行 |
| パイプラインバッチのストール | 2セッション以上進捗なし | ブロック済みとして浮上 |

### Reweave哲学

後ろ向きパスはリンクを追加するだけではない。**「今日この知見を書いたなら、何が違うだろうか？」** と問うことです。

知見は完成した成果物ではなく、生きたドキュメントです。先月書いた知見は先月の理解で書かれた。その後：新しい知見が存在し、理解が深まり、クレームのシャープ化が必要かもしれない。Reweaveは現在の知識に基づいて知見を完全に再考することです。

## システム自己進化

システムはフリクションシグナル（observations、tensions）を蓄積し、定期的にそれらをレビューして自己進化します。

### Observations（ops/observations/）
作業中にフリクション、サプライズ、プロセスギャップ、方法論的洞察に気づいたら、即座に `ops/observations/` に原子的ノートとしてキャプチャする。各observationは散文タイトルとカテゴリ（friction | surprise | process-gap | methodology）を持つ。

### Tensions（ops/tensions/）
2つの知見が互いに矛盾するとき、または実装が方法論と衝突するとき、即座にテンションを `ops/tensions/` にキャプチャ。各テンションは競合する知見を名前で挙げ、解決状況を追跡（pending | resolved | dissolved）。

### 蓄積トリガー
- **pending observationが10件以上** → /arscontexta:rethink でトリアージ・処理
- **pending tensionが5件以上** → /arscontexta:rethink で競合を解決

## タスク管理

### 処理キュー（ops/queue/）
パイプラインタスクはJSONキューで追跡される。各知見は `create → reflect → reweave → verify` フェーズを経るキューエントリーを1つ持つ。

### メンテナンスキュー
メンテナンス作業は同じキューにパイプライン作業と並んで存在する。/next が各呼び出し時に条件を評価：発動した条件は `type: "maintenance"` キューエントリーを作成し、満たされた条件は自動クローズ。

| コンテンツタイプ | 保存先 | 例 |
|--------------|------|---|
| 技術的知見・洞察 | notes/ | 論文クレーム、パターン、アンチパターン |

## パイプライン遵守

**notes/に直接書かない。** すべてのコンテンツはパイプラインを通じてルーティングされる: `inbox/ → /reduce → notes/`。/reduceを実行せずに notes/ にファイルを作成しようとしていることに気づいたら、STOP。まず `inbox/` を通じてルーティングする。パイプラインは直接書き込みが品質ゲートをスキップするために存在する。

フルオートメーションはデイワンからアクティブ。すべての処理スキル、すべての品質ゲート、すべてのメンテナンスメカニズムが即座に利用可能。バルトのサイズが一定規模になるまで待つ必要はない。

## 自己改善

フリクションが発生したとき（検索が失敗、コンテンツを間違った場所に配置、ユーザーが修正、ワークフローが壊れる）：
1. /arscontexta:remember を使って `ops/observations/` にobservationとしてキャプチャ — またはセッションキャプチャが自動的にトランスクリプトから検出
2. 現在の作業を続ける — 脱線しない
3. 同じフリクションが3回以上発生したら、このコンテキストファイルの更新を提案
4. ユーザーが「これを覚えておいて」または「常にXをして」と明示的に言ったら、即座にこのコンテキストファイルを更新

何か新しいものを作るとき：
- 未来のエージェントがこれを見つけられるか？（discovery-first）
- これはどんなメンテナンスが必要か？（持続可能性）
- 何が問題になり得るか？（失敗モード認識）

## 自己拡張ブループリント

このシステムはあなた自身で拡張できる。

### 新しいスキルの構築
`.claude/skills/skill-name/SKILL.md` を作成：
- YAMLフロントマター（name、description、allowed-tools）
- スキルが何をするかの指示
- 品質ゲートと出力フォーマット

### フックの構築
`.claude/hooks/` にイベントで起動するスクリプトを作成：
- `SessionStart`: セッション開始時にコンテキストを注入
- `PostToolUse (Write)`: 作成後にノートを検証
- `Stop`: 終了前にセッション状態を保存

### スキーマの拡張
テンプレートにドメイン固有のYAMLフィールドを追加。ベースフィールド（description、type、created）は普遍的。あなたのユースケースのためにクエリ可能なフィールドを追加。

### MOCの拡張
MOCが〜35件を超えたら、分割する。親MOCにリンクするサブMOCを作成。階層はコンテンツから出現する、計画からではない。

## よくある落とし穴

### コレクターの誤謬
Xやブログを溜め込むだけで抽出しない。inboxが増え続け処理が追いつかなくなったら、キャプチャを止めて抽出を始めること。WIP上限: 現在あるものを処理してから追加する。

### 言葉通りのコピー
ソースを要約することは知見を抽出することではない。各ノートは素材を変換しなければならない — あなたの言葉、あなたのフレーミング、あなたの論証。タイトルが「この知見が主張するのは [タイトル]」と機能しないなら、まだ知見ではない。

### 孤立ノート
接続のない知見は二度と見つからない知見だ。すべての知見は少なくとも1つのテーママップリンク（topicsフッター）と、理想的には関連知見へのインラインリンクが必要。孤立ノートを発見するためにヘルスチェックを実行する。

### 生産性ポルノ
知識の蓄積よりシステムの改善を続けることは誘惑的だ。バルトは研究に奉仕し、その逆ではない。方法論より知見に費やす時間が少なくなったら、再調整する。100件の知見が貯まるまでシステム変更は最小限に。

## システム進化

このシステムはリサーチ設定でシードされました。使用を通じて進化します。

### 予期される変化
- **スキーマ拡張** — まだテンプレートにないが追跡する価値のあるフィールドを発見するだろう。真のクエリニーズが生じたときに追加する。
- **MOC分割** — トピック領域が〜35件を超えたら、親にリンクするサブMOCに分割する。
- **処理の改良** — 処理サイクルにパターンが発達する。繰り返しパターンを `ops/methodology/` の方法論更新としてエンコードする。
- **新しいノートタイプ** — 知見ノートとMOCを超えて、テンションノート（矛盾のため）、方法論ノート（パターンのため）、または合成ノート（高次クレームのため）が必要かもしれない。

### フリクションのサイン（これらに行動する）
- 接続なしにノートが蓄積 → 接続探索の頻度を上げる
- 存在することが分かっているものが見つからない → セマンティック検索またはより多くのMOC構造を追加
- 誰もクエリしないスキーマフィールド → 削除（スキーマは検索に奉仕、官僚制に奉仕しない）
- 処理が形式的に感じる → サイクルを単純化するか機械的な部分を自動化

### 導出の根拠
- 粒度: 原子的 — 技術概念の横断的比較のため（論文×ブログ×実験）
- 処理: 重い — 「本質的な情報を抽出」「深堀」「アウトプット」シグナルから
- ナビゲーション: 3階層 — 年間100〜200件の知見 × 原子的粒度
- セルフスペース: 無効 — 知識グラフが主役、`ops/` に統合
- 自動化: フル — Claude Codeプラットフォームがサポート
