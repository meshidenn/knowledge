# 技術知見システムの導出根拠

このシステムは2026-03-24に以下の会話シグナルから導出されました。

## 核心的な要件

ユーザーは「日々の情報収集の結果を構造化して、本質的な情報を抽出したい」と述べました。情報源は論文・ブログ記事・Xのpost。目的は自分の理解を深めることと、特定技術の検証（アウトプット予定）です。

## ディメンション選択の根拠

### 粒度: 原子的 (Atomic)

「本質的な情報を抽出したい」という直接的なシグナルから。単一ソースへの要約では、論文・ブログ・実験の横断的な比較ができない。原子的な知見ノートは構成可能なノードを作り、Wikiリンクによるグラフ的接続を可能にする。

技術ドメインでは同じコンセプトが異なるソースで異なる語彙で表現される（例：「residual connections」vs「skip connections」）。原子的粒度はこの横断的比較を最も効果的にする。

### 処理: 重い (Heavy)

3つの収束シグナル：
1. 「本質的な情報を抽出」— 抽出パイプラインが必要
2. 「深堀もします」— 単なるキャプチャ以上の処理が必要
3. 「アウトプットしたい」— 出力指向の処理は深い分析を要求

重い処理は：reduce（ソースから知見を抽出）→ reflect（接続を発見）→ reweave（古いノートを更新）→ verify（品質確認）の完全なパイプラインを意味する。

### ナビゲーション: 3階層

年間ボリューム推定：
- 論文: 週1〜2本 × 約50週 = 約100本 → 各論文から複数の知見 → 年間200〜400件の知見
- ブログ: 週10本 × 処理対象10% = 年間50本
- Xのpost: 多数 × 重要なもの5% = 年間適量

合計: 年間100〜200件の知見。原子的粒度と組み合わせると3階層ナビゲーション（ハブ→ドメインMOC→トピックMOC→知見）が必要。2階層では100件以上でナビゲーションが崩壊する。

### スキーマ: モデレート (Moderate)

複数のソースタイプ（paper/blog/x-post/experiment）と `status: output-ready` フラグが必要。技術ドメインの構造的な性質は中程度のスキーマを正当化するが、稠密なスキーマは捕捉フリクションを引き起こす。

### 自動化: フル (Full)

Claude Codeプラットフォームがフックとスキルを完全にサポート。重い処理ニーズ + フルオートメーション = 持続可能なシステム。

### セルフスペース: 無効

Researchプリセットのデフォルト。知識グラフがフォーカス。エージェントのアイデンティティはCLAUDE.mdに直接組み込まれ、目標はops/goals.mdにルーティングされる。

## 一貫性の検証

3つのハード制約をチェック — 違反なし：
- atomic + 2-tier + volume > 100: 3-tier を選択したため回避
- full automation + platform無サポート: Claude Codeが完全にサポート
- heavy processing + manual + no-pipeline: フルオートメーションが解決

ソフト制約：
- atomic + explicit linking のみ（semantic searchなし）: 明示的リンクで現在のボリュームには十分。qmdが将来必要になった場合に有効化可能。
